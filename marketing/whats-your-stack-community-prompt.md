# Stackfax What’s Your Stack Community Prompt

This file contains early community prompt ideas for Stackfax.

The goal is to create a simple way for builders, noobs, experts, and businesses to describe their Ai stacks without turning the conversation into hype, flexing, or unsafe oversharing.

## Core Idea

What’s your stack?

Better:

What’s your stack, and what job is it built for?

Stackfax version:

Stackfax your stack before you copy someone else’s.

## Why This Matters

People are already comparing Ai stacks in public.

They ask:

- Which model should I use?
- Which subscription should I pay for?
- Should I buy a Mac mini?
- Should I use local models?
- Should I use OpenClaw?
- Can I avoid API bills?
- Should I copy this creator stack?
- What are other builders using?

But most public stack talk is incomplete.

A useful stack prompt should capture:

- Use case
- Tools
- Models
- Hardware
- Hosting
- Workflow
- Permissions
- Approval gates
- Cost concerns
- Risk flags
- What worked
- What broke
- What changed

## Community Prompt 1

What’s your Ai stack, and what job is it built for?

Share:

- Main use case
- Tools/models
- Local, cloud, or hybrid
- Hardware
- What agents can touch
- What requires human approval
- Biggest cost risk
- Biggest safety risk
- What you would change if starting over

## Community Prompt 2

What’s your OpenClaw stack?

Share:

- What you’re building
- Model setup
- Local or cloud
- Ollama or paid APIs
- Communication channel
- Memory setup
- Tools/skills
- Approval gates
- Biggest issue so far
- Best thing that works

## Community Prompt 3

What’s your noob-safe Ai stack?

Share the simplest setup you would recommend to someone starting today.

Include:

- One primary model
- One workflow
- One safety rule
- One thing to avoid
- One upgrade path
- One recheck trigger

## Community Prompt 4

What stack would you not copy again?

Share:

- What looked good at first
- What broke
- What cost too much
- What was overkill
- What you replaced
- What you would do now

## Community Prompt 5

Stack of the Month nomination:

Share a stack that is useful, not just flashy.

Include:

- Who it is for
- What it does well
- What it should not be used for
- Cost range
- Risk flags
- Approval gates
- Why others can learn from it

## Stackfax Reply Format

When replying to a shared stack, Stackfax-style feedback should focus on fit.

Useful response format:

- Best fit:
- Possible overkill:
- Main risk:
- Missing guardrail:
- Best next move:
- Recheck trigger:

## Example Response

Best fit:

Good starter OpenClaw stack for learning local agents.

Possible overkill:

Buying dedicated hardware before the first workflow is defined.

Main risk:

Permission boundary is unclear.

Missing guardrail:

Human approval before sending messages or touching files.

Best next move:

Start with one observe-and-summarize workflow.

Recheck trigger:

Recheck after adding Discord, email, API credentials, or scheduled tasks.

## Safe Sharing Rule

Do not ask people to share:

- Passwords
- API keys
- Recovery codes
- Payment details
- Wallet information
- Private customer data
- Private legal documents
- Private financial account information
- Sensitive personal information

Community stack sharing should be public-safe.

## Public Stack Fields

Safe public fields may include:

- General use case
- Tool names
- Model names
- Local/cloud/hybrid direction
- Hardware category
- Communication channel category
- General workflow
- Approval gate status
- Cost range
- Risk flags
- Lessons learned

Unsafe public fields may include:

- Private customer workflows
- Exact credentials
- Private files
- Private screenshots
- Internal URLs
- Payment data
- API keys
- Wallet details
- Sensitive business data

## Stackfax Community Principles

Be useful.

Do not shame beginners.

Do not worship complexity.

Do not copy stacks blindly.

Do not flex expensive setups without explaining the job.

Do not call a stack good without naming the use case.

Do not confuse hype with fit.

Do not treat local as automatically better.

Do not treat cloud as automatically better.

Do not automate before approval gates exist.
