# Consultant Wedge Gold Signals

This file captures early public scouting signals for the Stackfax consultant wedge.

The goal is to turn scattered field research into durable Stackfax doctrine.

## Core Thesis

Stackfax can help Ai consultants, agencies, automation sellers, and business operators turn messy Ai interest into safer, clearer implementation decisions.

The market signal is not just:

- people want Ai
- people want agents
- people want automation

The stronger signal is:

- businesses are building Ai workflows
- many workflows are half-built or operationally homeless
- agents are moving toward real actions
- testing, ownership, approval gates, and receipts are weak
- consultants need safer pre-build checks
- Stackfax can become the verdict layer before implementation

## Signal 1: Founder Ai Stack Audits

### Lane

Consultant wedge  
Business Ai audit  
Workflow fit  
Production readiness

### Pattern

Founders and businesses are already using Ai tools, agents, automations, and workflow builders.

But many setups are fragmented:

- half-built agents
- orphan automation flows
- scattered tools
- no owner
- no success metric
- no clear workflow home
- no shipped operating system

### Stackfax Takeaway

Technically built is not the same as operationally housed.

A workflow is not ready just because the automation runs.

Before building another tool, the business should answer:

- What process does this belong to?
- Who owns it?
- What data comes in?
- Where does the output go?
- What human action does it trigger?
- What metric proves it worked?
- What happens when it fails?

### Stackfax Doctrine

A workflow without an owner, data path, and win condition is not in progress.

It is in purgatory.

### Consultant Wedge

This gives consultants a safer first move:

Do not sell another automation first.

Audit the existing stack, identify what is homeless, kill weak workflows, and only then recommend a pilot.

## Signal 2: Agent Testing Before Shipping

### Lane

Agent testing  
Production readiness  
Governance  
Run receipts  
Non-engineer evals

### Pattern

Many teams are deploying agents without repeatable testing.

Common weak patterns include:

- ship and wait for complaints
- manually check a spreadsheet when someone remembers
- no golden examples
- no regression checks
- no pass/fail threshold
- no launch gate
- no receipt showing what version, model, prompt, or tool route was tested

### Stackfax Takeaway

Production testing cannot be “ship it and watch Slack.”

Non-engineering teams do not always need research-grade evals first.

They need a minimum launch gate they will actually use.

### Minimum Launch Gate

A basic agent-readiness check should include:

- 20 to 30 golden examples
- clear pass/fail rules
- edge cases from real workflow history
- a threshold where launch gets blocked or reviewed
- a receipt showing what version, model, prompt, and tools were tested
- a re-run when prompts, tools, or model routes change

### Stackfax Doctrine

If a team cannot tell whether the current agent is better, worse, or just different than the last one, they do not have an eval layer.

### Consultant Wedge

Before a consultant puts an agent in front of customers, Stackfax can check whether the agent has:

- test cases
- approval gates
- failure handling
- run receipts
- launch thresholds
- recheck triggers

This makes the consultant look safer and gives the buyer more confidence.

## Signal 3: Autonomous Business Agents

### Lane

Autonomous agents  
Real-money automation  
Approval gates  
Runtime control  
Decision receipts

### Pattern

Business agents are moving from demos into real workflows:

- storefronts
- ad management
- lead generation
- cold email
- CRM updates
- analytics
- checkout
- customer-facing workflows

The dangerous failure mode is not only wrong output.

It is confident wrong action in a novel condition.

### Stackfax Takeaway

Benchmarks test known scenarios.

Production creates unknown scenarios.

An agent can sound confident exactly when the situation is unfamiliar.

### Stackfax Questions

A serious agent-readiness review should ask:

- What can the agent execute without approval?
- What actions move money?
- What actions contact customers?
- What actions change spend?
- What actions update CRM, inventory, checkout, or production systems?
- What detects novelty?
- What happens when confidence is wrong?
- What receipt shows the decision chain?
- What actions are reversible?
- What actions require human approval?

### Stackfax Doctrine

Autonomy should route by consequence, not confidence.

Confidence is not a control layer.

The system needs evidence of familiarity, not just confidence about the answer.

### Consultant Wedge

A consultant can use Stackfax before selling full autonomy:

Map action classes first.

Then pilot the workflow.

Then add review gates.

Then track receipts.

Then recheck before scaling.

## Signal 4: Contractor Floorplan Estimate Tool

### Lane

Vertical Ai tool  
Boring-money workflow  
Professional validation  
Run receipts  
Approval gates

### Pattern

Vertical Ai tools are being built for real business workflows, such as contractors, architects, designers, builders, and homeowners using floorplans or construction drawings.

The workflow may include:

- 2D floorplan upload
- area approximation
- room breakdown
- wall/layout measurement
- material list
- rough bill of materials
- early cost-estimation support

### Stackfax Takeaway

This is a strong workflow because the input and output are clear.

But the risk boundary is also clear.

Wrong measurements or bad assumptions can affect quotes, procurement, client expectations, and real-world build decisions.

### Stackfax Questions

A vertical Ai workflow like this should show:

- source plan used
- assumptions made
- measurement confidence
- room/area breakdown
- material estimate basis
- what still needs human review
- what should not be used for final quote, procurement, or construction decisions yet

### Stackfax Doctrine

The value is not replacing the professional.

It is making the first-pass estimate reviewable.

### Consultant Wedge

This is the kind of workflow where Stackfax can identify:

- workflow fit
- professional validation requirements
- approval gates
- run receipt needs
- liability/risk boundaries
- production-readiness status
