Stackfax Quick Report Delivery Terms

This file defines the basic delivery terms for early Stackfax Quick Reports.

The goal is to make expectations clear before a user pays for or submits a report request.

⸻

Core Rule

Set expectations before taking money.

Stackfax is early, manual, and trust-first.

Do not overpromise.

Do not pretend the report is automated before it is.

Do not collect sensitive information Stackfax does not need.

⸻

Product

Stackfax Quick Report

A manual report that checks whether the user’s Ai tools, models, subscriptions, hardware, workflow, agents, automations, permissions, and cost risks match the task.

The report is practical stack guidance.

It is not a guarantee of performance, savings, safety, compliance, or business outcome.

⸻

Price

Starting early price:

$19

This price may change later as the report format improves.

Early pricing is for testing the report format, learning from real stacks, and making the delivery process repeatable.

⸻

Delivery Window

Expected delivery target:

24 to 48 hours after intake is received and accepted.

Delivery may take longer if:

* the intake is incomplete
* the stack is unclear
* follow-up questions are required
* the request is outside the current Stackfax scope
* the user submitted unsafe private information that needs to be removed
* the report type needs clarification
* the workflow involves business automation, customer data, credentials, payments, wallets, or production systems

If delivery is delayed, Stackfax should send a clear update when possible.

⸻

What The Buyer Gets

A Stackfax Quick Report may include:

* report type
* use case summary
* overall rating
* Stack Score
* main verdict
* risk flags
* relevant badges
* hardware verdict when relevant
* local vs cloud fit
* token burn review
* model subscription fit
* workflow fit notes
* approval gate warnings
* run receipt notes when relevant
* cost visibility notes when relevant
* best next move
* suggested upgrade path
* 30-day recheck notes

The report should answer the user’s main decision.

If the user still does not know what to do next, the report failed.

⸻

Supported Quick Report Types

Current supported report types include:

* Hardware Verdict
* Token Burn Audit
* OpenClaw Starter Stack Check
* Model Subscription Fit Check
* Business Automation Safety Check
* Vendor Stack Verdict
* General Stackfax Quick Report

Possible future or higher-touch report types include:

* Pro Report
* Executive Audit
* Team Stack Review
* Agent Governance Audit

Do not sell unsupported report types as finished products before they are ready.

⸻

Intake Requirement

The buyer must submit the Quick Report intake before the report can begin.

The intake should describe:

* user goal
* current stack
* tools and subscriptions
* hardware plans
* local, cloud, or hybrid preference
* workflow
* cost concerns
* risk concerns
* automation comfort level
* approval gates
* desired report outcome
* what decision the user needs help making

A good intake should help answer:

* What are you trying to build or improve?
* What are you using now?
* What feels confusing, expensive, risky, or overbuilt?
* What should the system be allowed to do?
* What should require human approval?
* What decision should the report help you make?

⸻

Safety Rule

Do not submit:

* passwords
* API keys
* recovery codes
* payment details
* wallet information
* seed phrases
* private keys
* private customer data
* private legal documents
* private medical documents
* private financial account information
* sensitive personal information
* identity documents
* employee records
* client confidential files
* private inbox exports
* source code secrets
* production credentials

Stackfax only needs a description of the stack, workflow, tools, goals, risks, and concerns.

Core principle:

Stackfax reviews the stack problem, not the user’s private life.

⸻

If Sensitive Information Is Submitted

If a user submits sensitive information:

1. Do not use it.
2. Do not copy it into the report.
3. Do not store it in the public repo.
4. Ask the user to remove it.
5. Request a cleaned-up intake.
6. Continue only after the intake is safe.

Stackfax may pause the report until the intake is cleaned up.

⸻

What Stackfax Does Not Provide

Stackfax does not provide:

* legal advice
* financial advice
* trading advice
* medical advice
* security certification
* compliance certification
* guaranteed cost savings
* guaranteed business outcomes
* guaranteed automation safety
* guaranteed model performance
* guaranteed revenue improvement
* done-for-you implementation
* emergency support
* managed IT support
* production deployment
* account access
* wallet access
* credential handling

Stackfax can identify risks, workflow fit, approval gate needs, cost concerns, and practical next steps.

Stackfax does not guarantee outcomes.

⸻

Follow-Up Questions

Stackfax may ask follow-up questions if:

* the stack is unclear
* the report type is unclear
* the user’s main decision is unclear
* the request involves business automation
* the request involves customer data
* the request involves credentials
* the request involves payments, wallets, or financial accounts
* the intake is missing important context
* the workflow is too broad to evaluate clearly
* the submitted information includes sensitive data that must be removed

If the user does not respond to follow-up questions, the report may be paused.

⸻

Refund Expectation

If Stackfax cannot deliver the report, the buyer should receive a refund.

If the intake is incomplete and the user does not respond to follow-up questions, the report may be paused until the required information is provided.

Delivered reports are custom/manual and refund requests may be reviewed case by case.

Possible refund reasons:

* Stackfax cannot complete the report as submitted
* the request is outside current scope
* the request requires unsafe access or sensitive data
* the buyer submitted information that cannot be safely reviewed
* Stackfax cannot deliver within a reasonable updated window

Refund handling should stay simple, fair, and trust-first.

⸻

Early Manual Status

Stackfax Quick Reports are early and manual.

This means:

* a human reviews the intake
* a human prepares the report
* report structure may improve over time
* feedback may improve future templates
* reports may vary as the format is refined
* reports are guidance, not guarantees

Do not pretend the report is fully automated.

Manual review is part of the early product.

⸻

Delivery Format

Early reports may be delivered as:

* email
* plain-text report
* markdown-style report
* Google Doc
* PDF later
* other approved delivery method

Recommended early delivery:

Email with a clean, plain-text or markdown-style report using simple headings.

The report should be readable without special software.

⸻

Feedback

Stackfax may ask for optional feedback after delivery.

Useful feedback includes:

* Was the verdict clear?
* Was the next move useful?
* What part helped most?
* What felt confusing?
* Would you want a deeper Pro Report?
* What should Stackfax check next time?
* Did the score feel fair?
* Did the risk flags feel accurate?
* Did the report answer your real decision?

Do not use feedback as a testimonial without clear permission.

⸻

Trust Rules

Do not include:

* fake urgency
* fake scarcity
* fake testimonials
* guaranteed savings
* guaranteed outcomes
* legal guarantees
* financial guarantees
* compliance guarantees
* security guarantees
* pressure to upgrade
* claims that Stackfax removes all risk
* claims that Stackfax is fully automated before it is

Every customer-facing promise should be small enough to keep.

⸻

Public / Private Boundary

Do not store real customer reports, emails, intake details, payment details, or private submissions in the public GitHub repo.

The public repo may include:

* blank templates
* fictional examples
* sample scenarios
* public doctrine
* anonymized product lessons
* public report structures

Private customer work should stay private.

⸻

Final Rule

Clarity before you build.

Trust before scale.

Manual first.

Repeatable second.

Automated later.
