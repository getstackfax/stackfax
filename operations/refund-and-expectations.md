# Stackfax Refund And Expectations Policy

This file defines early refund, delivery, and expectation rules for Stackfax reports.

The goal is to keep Stackfax honest, simple, and trust-first while reports are still manual.

---

## Core Rule

Stackfax is a trust product.

Do not overpromise.

Do not pretend the report is automated before it is.

Do not guarantee:

- savings
- outcomes
- revenue
- security
- compliance
- legal results
- trading results
- technical performance
- automation safety
- model performance

The report should give a practical stack-fit verdict, not a guarantee.

---

## Early Manual Status

Stackfax reports are early and manual.

This means:

- a human reviews the intake
- a human prepares the report
- reports may vary as the format improves
- feedback may improve future report templates
- the product is still being shaped

Do not imply that reports are fully automated until that is true.

---

## Delivery Window

Expected delivery target:

```text
24 to 48 hours after intake is received and accepted.

Delivery may be delayed if:

* the intake is incomplete
* the request is unclear
* the user submits unsafe private information
* follow-up questions are required
* the report type is outside the current Stackfax scope
* the request needs to be narrowed before review

Do not promise instant delivery unless the system can actually support it.

⸻

What Stackfax Provides

A Stackfax report may provide:

* overall rating
* Stack Score
* main verdict
* risk flags
* relevant badges
* hardware verdict
* token burn review
* model subscription fit
* workflow fit notes
* approval gate warnings
* run receipt notes
* best next move
* suggested upgrade path
* 30-day recheck notes

The exact sections may vary by report type.

⸻

What Stackfax Does Not Provide

Stackfax does not provide:

* legal advice
* financial advice
* medical advice
* trading advice
* tax advice
* security audit certification
* compliance certification
* guaranteed cost savings
* guaranteed business outcomes
* guaranteed automation safety
* guaranteed model performance
* done-for-you implementation
* emergency support

Stackfax gives a practical verdict layer.

The user remains responsible for final decisions.

⸻

Safety Rule

Users should not submit:

* passwords
* API keys
* recovery codes
* payment details
* wallet information
* private customer data
* private legal documents
* private medical information
* private financial account information
* sensitive personal information
* credentials
* internal URLs
* private screenshots
* confidential business records

If sensitive information is submitted, Stackfax may ask the user to remove it and resubmit the intake.

Stackfax should review stack fit and workflow risk without collecting secrets.

⸻

Refund Rule

Early refund rule:

If Stackfax cannot deliver the report, the buyer should receive a refund.

If the user submits an incomplete intake and does not respond to follow-up questions, Stackfax may pause the report until the needed information is provided.

If the report has already been delivered, refund requests may be reviewed case by case.

The refund policy should protect trust without encouraging unclear or unsafe intakes.

⸻

Simple Public Refund Language

Suggested public wording:

If Stackfax cannot deliver your report, you get a refund.

If your intake is missing key details, Stackfax may ask follow-up questions before starting.

Because reports are manual and custom, delivered reports are generally reviewed case by case for refund requests.

⸻

Expectations For Buyers

Buyers should understand:

* Stackfax is early
* reports are manual
* reports are based on the information provided
* recommendations are practical, not guaranteed
* scores may evolve as the standard improves
* the report is not professional legal, financial, medical, trading, tax, compliance, or security advice
* the user remains responsible for final decisions

A Stackfax report can help the user think through the stack.

It does not replace professional review where professional review is required.

⸻

Follow-Up Questions

Stackfax may ask follow-up questions if:

* the report type is unclear
* the stack is not described enough
* the workflow is too broad
* the user asks about sensitive systems
* the request involves business automation
* the request involves customer data
* the request involves payments, credentials, wallets, or financial accounts
* the intake includes unsafe information
* the request appears outside current Stackfax scope

Follow-up questions should make the report safer and more useful.

They should not be used to pressure the user into a bigger purchase.

⸻

Report Refusal Or Pause

Stackfax may refuse or pause a report if the request requires:

* handling passwords
* handling API keys
* handling payment details
* accessing wallets
* accessing financial accounts
* reviewing private legal documents
* processing sensitive customer data
* giving professional advice
* unsafe automation instructions
* misleading or fraudulent use
* instructions to bypass safeguards
* instructions to hide or misrepresent Ai use

If a report is paused or refused, explain the reason plainly.

If payment was taken and the report cannot be delivered safely or usefully, issue a refund according to the current refund rule.

⸻

Early Buyer Bonus

Early paid users may receive a beta preview of the upcoming Stackfax Pro Report format.

This is:

* a beta preview
* a bonus
* a way to show deeper report value
* a way to help shape the Stackfax standard

This is not:

* a finished Pro product
* a guaranteed upgrade
* a fake discount
* a guarantee of future pricing
* a promise that all future Pro features will be included

Describe early buyer bonuses honestly.

Do not create fake scarcity.

Tone Rule

Keep refund and expectation language plain.

Avoid scary legal language unless needed later.

The goal is clarity, not intimidation.

⸻

Final Rule

Set expectations before taking money.

Trust is easier to protect than rebuild.
