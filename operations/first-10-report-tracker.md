# Stackfax First 10 Report Tracker

This file defines how Stackfax should track the first 10 real or beta reports.

The goal is to learn from early users without storing sensitive customer data in the public repo.

## Core Goal

Track the first 10 Stackfax reports by pattern, report type, verdict, and lesson learned.

Do not track private customer details in the public repo.

## Public Repo Safety Rule

This tracker should not include:

- Real full names
- Real emails
- Payment details
- Passwords
- API keys
- Private customer data
- Private business data
- Private legal documents
- Private financial information
- Wallet information
- Sensitive screenshots
- Real customer reports

Use anonymized labels only.

Example:

- Report 01
- Beta user 01
- Stackbuilder 01
- Hardware-curious user 01
- Token-burned user 01

## What To Track

For each early report, track:

- Report number
- Date
- User type
- Main pain point
- Report type
- Main verdict
- Score range
- Risk flags
- Best next move
- Was it free or paid?
- Was feedback received?
- Product lesson
- Template improvement

## Report Status Options

Use simple statuses:

- Not started
- Intake received
- Needs follow-up
- In progress
- Delivered
- Feedback received
- Archived

## Product Lessons To Watch

Look for patterns such as:

- Which report type gets requested most?
- Which intake questions confuse users?
- Which verdicts are most useful?
- Which risk flags appear repeatedly?
- Which templates are hardest to fill?
- Which report sections users value most?
- Which users might pay for Pro?
- Which users only need free mini reports?
- Which pain points deserve their own product page?

## Feedback Questions

After delivery, ask:

- Was the verdict clear?
- Was the best next move useful?
- What part helped most?
- What felt confusing?
- Would you pay for a deeper Pro Report?
- What should Stackfax check next time?

## Success Signals

Early reports are working if users say:

- The next move is clear
- The report prevented a bad purchase
- The report reduced subscription confusion
- The report explained token burn
- The report clarified OpenClaw setup
- The report helped set approval gates
- The report was worth paying for
- The user wants a deeper version

## Failure Signals

Early reports need improvement if users say:

- The report was too generic
- The verdict was unclear
- The score felt arbitrary
- The next move was not actionable
- The intake took too long
- The report missed the real question
- The report did not feel worth paying for

## Private Tracker Note

A private version of this tracker may include customer email, payment status, delivery links, and report files.

The public repo should only track anonymized product lessons.

## Final Rule

The first 10 reports are not just sales.

They are training ground for the Stackfax standard.
