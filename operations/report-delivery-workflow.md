# Stackfax Report Delivery Workflow

This file defines the early manual workflow for delivering Stackfax reports.

The goal is to make report delivery simple, honest, repeatable, and safe before building a full app or automation system.

## Core Goal

Turn a paid or free Stackfax intake into a useful report delivered within a clear delivery window.

Stackfax reports should help the user understand:

- what fits
- what is risky
- what may be overbuilt
- what should require approval
- what the best next move is
- what should be rechecked later

Manual delivery comes first.

Automation comes later.

---

## Delivery Window

Expected delivery target:
```text

24 to 48 hours after intake is received and accepted.

This window starts after Stackfax has enough information to review the stack.

If the intake is incomplete, unclear, unsafe, or missing core details, request clarification before starting the delivery clock.

---

## Intake Sources

Stackfax may receive report requests from:

* Free Mini Report intake form
* Gumroad Quick Report purchase
* direct user request
* partner / consultant referral
* future paid business audit request

Every request should be treated as untrusted until reviewed.

---

## First Review Checklist

Before preparing a report, check:

* Is this a free Mini Report or paid Quick Report?
* Is there a usable email/contact method?
* What is the user trying to build, buy, automate, or deploy?
* What tools, models, subscriptions, agents, workflows, or hardware are involved?
* Is the user asking for legal, financial, medical, security, trading, or compliance guarantees?
* Did the user submit passwords, API keys, payment details, wallet information, private customer data, private legal/financial documents, or sensitive personal information?
* Is the request safe to answer as stack guidance?

If sensitive information was submitted, do not use it in the report. Tell the user not to submit that type of information and ask for a safer description instead.

---

## Free Mini Report Workflow

Use this for free submissions.

Output

A Free Mini Report should include:

* basic verdict
* 1–5 star rating
* 2–3 risk flags
* what fits
* what looks overbuilt
* best next move
* recheck trigger

Goal

The Free Mini Report should be short, useful, and clear.

It should not try to solve the user’s entire stack.

It should give enough value that the user understands what Stackfax does.

---

## Quick Report Workflow

Use this for paid Gumroad orders.

Output

A Quick Report may include:

* overall verdict
* Stack Score
* main risk flags
* relevant badges
* hardware verdict
* token burn review
* model/subscription fit notes
* workflow fit notes
* approval gate warnings
* best next move
* 30-day recheck notes

Goal

The Quick Report should feel more complete than the Free Mini Report.

It should give a practical next-step plan without pretending to guarantee outcomes.

---

## Response Priority

Paid Quick Reports should be prioritized before free Mini Reports.

Suggested order:

1. Paid Quick Report with complete intake
2. Paid Quick Report needing clarification
3. Free Mini Report with complete intake
4. Free Mini Report needing clarification
5. speculative or unsafe requests

---

## First Response Message

Use this when confirming a received intake.

Thanks for submitting your Stackfax intake.

I received your stack details and will review them for a Stackfax report.

Please do not send passwords, API keys, payment details, wallet information, private customer data, private legal documents, private financial documents, or sensitive personal information.

Stackfax gives stack guidance, not guaranteed business, financial, legal, trading, medical, security, compliance, or technical outcomes.

For paid orders, add:

Your Quick Report target is 24–48 hours after accepted intake.

Clarification Message

Use this when the intake is too vague.

Thanks for the Stackfax intake.

I need one or two clarifications before preparing the report:

1. What are you trying to build, buy, automate, or deploy?
2. What tools, models, subscriptions, hardware, or workflows are currently involved?
3. What decision are you trying to make next?

Please do not send passwords, API keys, payment details, wallet information, private customer data, private legal documents, private financial documents, or sensitive personal information.

---

## Safety Boundaries

Stackfax should not claim to guarantee:

* business results
* legal outcomes
* financial outcomes
* trading outcomes
* medical outcomes
* security outcomes
* compliance outcomes
* technical performance

Stackfax should avoid directly handling:

* passwords
* API keys
* payment details
* wallet information
* private customer data
* private legal documents
* private financial documents
* sensitive personal information

Stackfax should frame outputs as:

* stack guidance
* workflow review
* risk flags
* fit assessment
* suggested next steps
* recheck recommendations

## Completion Rule

A report is complete when:

* the verdict is clear
* the main risks are named
* the next move is specific
* unsafe claims are avoided
* sensitive information is excluded
* the delivery is recorded privately

The goal is not to sound impressive.

The goal is to help the user make a safer, clearer stack decision.
