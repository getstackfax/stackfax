# Stackfax Report Delivery Workflow

This file defines the early manual workflow for delivering Stackfax reports.

The goal is to make report delivery simple, honest, repeatable, and safe before building a full app or automation system.

## Core Goal

Turn a paid or free Stackfax intake into a useful report delivered within a clear time window.

## Delivery Window

Expected delivery target:

24 to 48 hours after intake is received and accepted.

This window may change later as volume, report depth, and product format evolve.

## Current Delivery Method

Early reports may be delivered by:

- Email
- Google Doc
- Markdown file
- PDF later
- Other approved delivery method

Recommended early method:

Email with a clean Markdown-style report.

## Basic Workflow

1. User pays or requests a report.
2. User completes the intake form.
3. Stackfax checks the intake for safety issues.
4. Stackfax confirms the report type.
5. Stackfax drafts the report manually.
6. Stackfax reviews the report for clarity and safety.
7. Stackfax delivers the report.
8. Stackfax requests optional feedback.
9. Stackfax uses lessons to improve templates and scoring.

## Intake Safety Check

Before starting a report, check whether the user submitted unsafe or private information.

The intake should not include:

- Passwords
- API keys
- Recovery codes
- Payment details
- Wallet information
- Private customer data
- Private legal documents
- Private financial account information
- Sensitive personal information

If unsafe information is included, do not use it.

Ask the user to resubmit with sensitive details removed.

## Report Type Confirmation

Confirm which report type best fits the request.

Possible report types:

- Free Mini Report
- Stackfax Quick Report
- Mac Mini Hardware Verdict
- Token Burn Audit
- OpenClaw Starter Stack Check
- Model Subscription Fit Check
- Business Automation Safety Check
- Vendor Stack Verdict

If the user is unsure, choose the best-fit report type based on the intake.

## Drafting Workflow

Use the relevant template or sample report.

Suggested drafting steps:

1. Identify the user goal.
2. List the current stack.
3. Choose the main report type.
4. Assign the main verdict.
5. Add rating and score.
6. Add risk flags.
7. Add relevant badges.
8. Review hardware fit.
9. Review token burn risk.
10. Review model subscription fit.
11. Review approval gates.
12. Review privacy and credential isolation.
13. Add best next move.
14. Add 30-day recheck guidance.
15. Add final recommendation.

## Report Review Checklist

Before delivering, check:

- Is the verdict clear?
- Is the next move specific?
- Are risk flags relevant?
- Are safety warnings included?
- Did the report avoid overpromising?
- Did the report avoid pretending to be legal, financial, security, trading, or compliance advice?
- Did the report avoid using private sensitive information?
- Is the language plain enough for the user?
- Is the report useful even if the user does not buy anything else?

## Delivery Email Structure

Suggested delivery structure:

Subject:

Your Stackfax Quick Report is ready

Body:

- Thank the user
- State the report type
- Include the main verdict
- Include the report or report link
- Mention any safety notes
- Ask for optional feedback
- Mention that Stackfax is early and improving

## Optional Feedback Questions

Ask:

- Was the verdict clear?
- Was the next move useful?
- What part was most helpful?
- What felt confusing?
- Would you pay for a deeper Pro Report?
- What should Stackfax check next time?

## Follow-Up Rule

Do not pressure users into a bigger report.

If there is a relevant next step, explain it clearly.

Possible next steps:

- Free Mini Report
- Quick Report
- Pro Report beta preview
- 30-day recheck
- Business Automation Safety Audit
- Token Burn Audit
- Model Subscription Fit Check

## Recordkeeping

Keep internal notes about:

- Report type
- Main pain point
- Verdict
- Risk flags
- What template was used
- What could improve the report format
- Whether the user gave feedback

Do not store sensitive private information in the public repo.

## Quality Standard

A Stackfax report should be:

- Clear
- Useful
- Honest
- Specific
- Safety-aware
- Easy to act on
- Matched to the user’s current level

## Final Rule

Manual first.

Repeatable second.

Automated later.

The goal is to learn what a good Stackfax report should be before automating the process.
