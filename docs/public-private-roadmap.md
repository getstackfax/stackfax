Stackfax Public Private Roadmap

This file defines how Stackfax should separate the public trust layer from the private business engine over time.

The goal is to build trust publicly while protecting customer data, paid workflows, internal operations, proprietary scoring logic, and future automation systems.

⸻

Core Strategy

Public trust layer.

Private business engine.

Stackfax should show enough to build trust and explain the standard.

Stackfax should keep enough private to build the business safely.

Core principle:

Show enough to build trust. Keep enough private to build the business.

⸻

Stackfax is not just a code project.

It includes:

* product positioning

* templates

Some of that can be public.

Some of that should stay private.

What Should Never Go Public

Do not put these in the public repo:

* passwords
* API keys
* recovery codes
* payment details
* wallet information
* seed phrases
* private keys
* customer submissions
* customer emails
* real customer reports
* private business data
* client data
* private screenshots
* internal report delivery links
* private OpenClaw configs
* private Rabble Scout outputs
* internal automation credentials
* production credentials
* private inbox content
* source code secrets
* customer support messages
* payment provider records

Default rule:

Public repo gets principles and samples. Private storage gets real people and real submissions.

Future OpenClaw And Rabble Scout Layer

OpenClaw and Rabble Scouts should stay private at first.

Rabble Scouts may eventually help with:

* public field research
* pattern detection
* stack confusion classification
* report type suggestions
* badge suggestions
* risk flag suggestions
* drafting internal research notes
* finding repeated public pain signals
* summarizing Gold Reports

Rabble Scouts should not:

* post publicly
* contact users
* access customer data
* access credentials
* modify files without approval
* touch payments
* touch wallets
* touch production systems
* access private customer folders
* deliver reports without human review

Rabble Scouts observe, summarize, and report.

They do not operate the business.

Public Release Rule

Before something goes public, ask:

* Does this build trust?
* Does this expose customer data?
* Does this expose business operations?
* Does this reveal too much of the paid engine?
* Does this help users understand Stackfax?
* Does this create unnecessary cloning risk?
* Does this need to be public right now?
* Is it clearly labeled if it is fictional, sample, or prototype content?
* Could this be useful publicly without exposing private logic?

If the answer is unclear, keep it private until reviewed.

Private Storage Rule

Real customer work should live outside the public repo.

Possible private storage:

* Private Google Drive
* Private Notion
* Private Airtable
* Private encrypted folder
* Private database later
* Private CRM later

Early recommendation:

Use a simple private folder and avoid storing sensitive information unless needed.

Do not store customer secrets.

Do not store unnecessary private data.

Do not let convenience become a data liability.

Open Public / Closed Engine Rule

Stackfax can be open enough to create trust without giving away the whole engine.

Public can explain:

* what Stackfax checks
* why the categories matter
* what safe examples look like
* what the report types are
* what the risk flags mean
* what the rating system means

Private should protect:

* how reports are generated at scale
* customer-specific logic
* paid report workflow details
* advanced scoring implementation
* proprietary prompt chains
* private report examples
* internal prioritization logic
* customer operations

Public standard.

Private engine.

⸻

Customer Data Rule

Customer data should be treated as private by default.

Before using any real user content publicly, confirm:

* what can be shown
* what must be anonymized
* whether the user gave permission
* whether the quote can be edited
* whether the report can be referenced
* whether the user wants to stay anonymous

No real testimonial, report, screenshot, or customer example should be public without permission.

Good public repo content:

* examples

* templates

* payment details

* secrets
