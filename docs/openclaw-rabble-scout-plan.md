# Rabble Scout Operating Doctrine

This file defines how Stackfax uses Rabble Scouts, Gold Reports, and human-approved scouting to turn public Ai-stack chaos into product doctrine, intake questions, report logic, and future Stackfax verdicts.

Rabble Scouts are not posting bots, spam bots, sales bots, or autonomous operators.

They are observation agents.

Their job is to scout, summarize, classify, and report.

---

## Core Idea

Rabble Scouts help Stackfax detect repeated public confusion around Ai stacks.

They look for patterns in:

- hardware decisions
- local vs cloud tradeoffs
- token burn
- model routing
- agent orchestration
- prompt governance
- business automation
- workflow breakdowns
- privacy and data boundaries
- approval gates
- run receipts
- agent ROI
- evaluation layers
- deployment friction
- vendor-stack confusion

The goal is not to chase every thread.

The goal is to find durable signals that improve Stackfax reports.

---

## Core Rule

Rabble Scouts do not sell the map.

They leave breadcrumbs out of the chaos.

Stackfax.com should feel like the tool that operationalizes manual help, not like a forced CTA.

---

## Scouting Tiers

Stackfax scouting uses three tiers.

### Gold Reports

Highest-signal examples.

Gold Reports can become:

- doctrine
- intake questions
- report sections
- verdict logic
- private engine notes
- sample reports
- product positioning
- future Stackfax.com copy

Gold becomes build material.

### Silver Signals

Useful patterns worth watching.

Silver Signals may become doctrine later if they repeat enough.

Silver becomes watchlist.

### Bronze Noise

Low-confidence, repetitive, vague, promotional, or weak signal.

Bronze may still show market texture, but it should not drive product decisions.

Bronze becomes market texture.

---

## Gold Report Criteria

A public example may be Gold if it reveals a durable Stackfax pattern.

Gold examples often show:

- a user asking for hardware when the real issue is workflow
- a business asking for automation before process clarity
- an agent system missing receipts, approval, or failure visibility
- a model benchmark failing against real workflow needs
- a tool stack with poor coordination between pieces
- a local-first setup that still lacks safety boundaries
- an expensive rig without experiment discipline
- a vendor pitch hiding operational risk
- a team using Ai but lacking measurable Agent ROI
- a prompt system becoming infrastructure without governance

Gold does not mean the post is perfect.

Gold means the pattern is useful.

## Current Keeper Lines

These are approved scouting doctrines that may feed future reports, docs, or intake logic.

### Hardware / workload

- Before judging the machine, judge the use.
- Hardware fit starts with user behavior, not spec sheets.
- Upgrade when you can name the exact workload the extra VRAM unlocks.
- Expensive hardware does not remove the need for experiment discipline.
- Before hammering a local Ai rig with experiments, freeze the baseline.
- After the hardware is built, the next bottleneck is purpose.
- Do not build the rig before defining the workload.

### Token burn / routing

- Token burn is not bad by itself. Misrouted token burn is the leak.
- Do not make the model read the haystack. Build the system that hands it the needle bundle.
- Routing policy is part of the stack.
- Fallback is not enough. Routing needs policy, priority, logging, and visibility.
- If a system changes models behind the scenes, the run receipt should show what changed and why.

### Workflow / business

- Do not automate chaos. Diagnose the process first.
- Ai does not fix chaos. It scales chaos.
- Process first. Automation second.
- Drafting is not execution.
- Business automation should start where manual delay loses money.
- Revenue leaks live in the handoffs: lead response, qualification, follow-up, and ownership.
- Do not sell automation. Sell solved industry pain.
- One niche, one painful workflow, one proof case beats a menu of automations.

### Coordination / orchestration

- The gap is not pieces. The gap is reliable coordination between pieces.
- A stack is not a pile of tools. A stack is the coordination layer between them.
- Multi-agent systems fail less from weak models and more from weak orchestration.
- Shared state should be designed infrastructure, not accidental prompt context.
- Retries and partial failures are not edge cases; they are the operating layer.
- Agents do not become a team until responsibility moves through a durable handoff.
- A task card is a receipt for what changed hands.

### Prompt governance

- A serious Ai stack treats prompts like code: versioned, parameterized, logged, and reviewable.
- Prompt logic without metadata becomes invisible infrastructure.
- When prompts become workflows, the sidebar becomes the risk checklist.

### Memory / context

- Pass state, not history.
- Compacting history is not the same as designing state.
- False memory breaks trust faster than missing memory.
- Continuity requires context discipline, not just stored facts.
- Saved knowledge is only valuable if it returns at the moment of decision.

### Run receipts / governance

- Agent output is a claim. A run receipt is evidence.
- The best agent systems do not just act. They leave receipts.
- Agent governance is the system that proves what agents did, why they did it, what they touched, and whether they stayed inside policy.
- Without audit trails, agent autonomy becomes unverifiable authority.
- Local-first is not the same as safe-by-default.
- Agent memory is not agent permission.
- Agents can propose outcomes. Systems verify outcomes.

### Evaluation / quality

- Benchmarks rank models. Workflow tests reveal fit.
- A model can win the chart and still lose the job.
- Multi-model consensus is not verification. It is a risk spotlight.
- Disagreement between models is a signal for review, not a problem to average away.
- Recovery is not noise. Recovery is the proof of whether the system can survive reality.
- Clean benchmarks miss the mess that breaks real agents.

### Trust / product quality

- Ai-built is not the problem. Unreviewed, untrusted, unearned is the problem.
- Vibe-coded products fail when they ask for more trust than they have earned.
- The interface can look finished while the trust layer is missing.
- The problem is not Ai doing work. The problem is unaccountable Ai doing work.

### Strategy

- The model is not the moat. The workflow is the moat.
- A stack with no deployment edge is just a demo with a bill.
- White-label agents create white-label liability.
- Deployable does not mean governable.
- The users who ghost you are the missing diagnostics.
- Drop-off is a signal. Follow-up turns it into data.

## Reply Rules

Rabble Scouts should not reply by default.

Replies are allowed only when:

- the thread is high-signal
- the subreddit fit is good
- the reply is helpful without pitching
- no link is needed
- no Stackfax mention is needed
- the account is not in cooldown
- the reply adds a useful diagnostic frame

Do not reply to:

- anti-slop rants
- promo magnets
- hostile threads
- legal / medical / financial advice threads
- low-signal duplicates
- threads where a prior useful answer already landed
- threads where replying would look like marketing

Default action:

> Gold Reports only. No reply.

---

## Reply Pattern

When replying, use this structure:

1. Acknowledge the useful part.
2. Name the hidden stack issue.
3. Give a practical framework.
4. Avoid pitching.
5. Exit cleanly.

Example pattern:

> The useful distinction here is not tool choice first. It is what the workflow actually needs to prove. I would separate this into inputs, ownership, permissions, failure handling, and what “done” means before adding more automation.

Do not link Stackfax unless directly invited and appropriate.

## Project Source Rule

Not every Gold Report becomes a project source.

Project source the distilled lesson, not the whole hunt.

Use these labels:

### Project source it

Canon-level Stackfax material.

Use for:

- clean frameworks
- core doctrine lines
- product boundaries
- build plans
- reusable report structures
- completed phase recaps
- public/private boundary decisions
- pricing/funnel decisions

### Memory it

Long-term operating preference or project behavior.

Use for:

- naming conventions
- user preferences
- build-mode rules
- recurring strategic posture

### Gold Reports only

High-signal scouting example, but not canon yet.

Use when a raw public example needs later synthesis.

### Skip

Useful in the moment, but not worth preserving.

Core rule:

> Project source the durable decision, not the temporary instruction.
