# Stackfax Recheck System

The Stackfax Recheck System is the doctrine for deciding when an Ai stack, report, verdict, badge, route, workflow, or automation should be reviewed again.

The goal is to keep Stackfax verdicts honest as models, tools, pricing, workflows, permissions, and user needs change.

⸻

Core Idea

Ai stacks expire.

A stack can fit the job today and need a recheck after the market, workflow, model, tool, cost, or risk profile changes.

Core rule:

Stackfax checks the fit now. Rechecks keep the fit honest later.

⸻

Core Question

Has anything changed that could make the original Stackfax verdict stale, incomplete, or unsafe?

If yes, the stack may need a recheck.

⸻

Why This Matters

Ai stack advice ages quickly.

A verdict can change when:

* new models launch
* pricing changes
* usage limits change
* OpenClaw updates
* local models improve
* tool integrations change
* model routers change behavior
* workflows expand
* agents gain new tools
* credentials get connected
* customer data enters the stack
* communication channels are added
* automation moves from test to production
* hardware is added or removed

A good report should not pretend to be permanent.

Stackfax verdict:

A stack verdict should age honestly.

⸻

What A Recheck Means

A recheck does not mean the original report was wrong.

It means the conditions changed enough to review whether the verdict still fits.

A recheck may confirm:

* keep current stack
* test before switching
* simplify
* add guardrails
* update route
* pause automation
* improve cost visibility
* change hardware verdict
* update approval gates
* move from test to pilot
* move from pilot to production candidate

The point is not constant rebuilding.

The point is controlled reassessment.

Model Drop Recheck

New models create confusion.

Users may ask:

* Should I switch?
* Should I cancel my current subscription?
* Should I use this for coding?
* Should I use this for agents?
* Should I route cheap tasks to it?
* Should I migrate from my current setup?
* Is it better than my current stack?
* Does it reduce token burn?
* Does it change local vs cloud fit?

Stackfax rule:

A new model is not automatically a new stack.

A recheck decides whether the change fits the job.

⸻

Pricing Recheck

Pricing changes can alter model routing and subscription fit.

Recheck when:

* API prices change
* subscription tiers change
* limits change
* free tiers shrink
* usage caps tighten
* router pricing changes
* a cheaper model becomes good enough
* a premium model becomes too expensive for routine work

Stackfax rule:

Cost changes can change the stack.

⸻

OpenClaw Update Recheck

OpenClaw updates can affect:

* setup friction
* migration
* local model handling
* model routing
* communication channels
* skills
* MCP servers
* gateways
* Docker support
* macOS support
* privacy options
* compaction
* stability
* scheduled tasks

A user pinned on an older version may need to decide:

* update now
* test in sandbox
* wait
* stay pinned
* migrate gradually

Stackfax verdict:

Stable enough to test is not the same as safe enough for production.

⸻

Workflow Change Recheck

A workflow should be rechecked when the job changes.

Examples:

* observe-only becomes draft-with-approval
* draft-with-approval becomes limited automation
* internal workflow becomes customer-facing
* manual workflow becomes scheduled
* small task becomes recurring process
* solo user becomes team workflow
* test data becomes production data

Stackfax rule:

Automation level changes should trigger a recheck.

⸻

Permission Recheck

A recheck is needed when agent authority expands.

Trigger recheck when the agent gains access to:

* files
* email
* calendar
* CRM
* customer data
* credentials
* payment tools
* wallets
* financial accounts
* inventory systems
* production systems
* public posting tools
* communication channels

Stackfax principle:

Permission should expand slower than capability.

⸻

Customer Data Recheck

Customer data changes the risk profile.

Recheck when the stack starts touching:

* names
* emails
* phone numbers
* addresses
* order history
* support messages
* CRM records
* payment-related information
* private notes
* client files
* internal business records

Stackfax rule:

Customer data is not generic context.

⸻

Communication Channel Recheck

Communication channels are part of the stack.

Recheck when the agent can use:

* email
* Slack
* Discord
* Matrix
* Mattermost
* WhatsApp
* Telegram
* SMS
* support chat
* CRM inbox
* website chat
* public posting tools

Adding a channel can create wrong-person, wrong-thread, wrong-workspace, or public/private confusion.

Stackfax rule:

The channel is part of the stack.

⸻

Hardware Recheck

Hardware decisions should be rechecked when:

* local workload becomes clearer
* token costs rise
* privacy isolation becomes important
* local models improve
* always-on workflows appear
* existing hardware struggles
* user considers a Mac mini or local Ai box
* hardware price changes
* cloud-first becomes expensive
* cloud-first is still enough

Stackfax rule:

Hardware should match the job, not the hype.

⸻

Migration Recheck

Migration should be rechecked before moving from one stack to another.

Recheck when switching:

* model provider
* local runner
* agent framework
* hosting setup
* communication channel
* subscription stack
* hardware path
* toolchain
* memory system
* automation platform

Stackfax verdict:

Migration is not progress unless the workflow gets better.

⸻

Production Recheck

Production movement always deserves a recheck.

Trigger recheck when:

* demo moves to pilot
* pilot moves to production
* workflow gains real users
* workflow touches customers
* workflow touches money
* workflow touches inventory
* workflow touches production systems
* workflow runs unattended
* workflow becomes scheduled

Stackfax rule:

One lucky run is not a system.

⸻

Failure Recheck

Failures should trigger review.

Examples:

* agent sent wrong message
* tool loop burned tokens
* customer data leaked
* workflow skipped records
* browser automation broke
* cost spiked
* model routed wrong
* approval was skipped
* production data changed incorrectly
* task completed without evidence
* run receipt missing

Stackfax principle:

A safe failure is one the operator can understand.

⸻

Recheck Windows

Suggested default windows:

Low-risk starter stack: 30–60 days
Quick Report: 30 days
Hardware Verdict: 30–90 days
Token Burn Audit: 14–30 days
Business Automation Safety Audit: 14–30 days
Production workflow: 7–30 days
After migration: 7–30 days
After major tool/model update: recheck immediately before scaling

The recheck window should match risk level.

Higher-risk stacks need shorter windows.

Possible freshness labels:

Freshness should never imply permanent safety.

⸻

StackChecked Recheck Rule

StackChecked should include:

* use case
* score
* verdict
* checked date
* recheck window
* risk flags
* scope limits

StackChecked means:

This stack was checked for this job, at this time, with these limits.

It does not mean:

* certified
* future-proof
* legally approved
* compliance approved
* security certified
* safe forever
* profitable
* production-ready by default

⸻

Recheck Decision Receipt

A recheck should leave a decision receipt.

Template:

Original verdict:
Checked date:
Recheck trigger:
What changed:
Evidence reviewed:
Risks changed:
Costs changed:
Permissions changed:
Workflow changed:
New verdict:
What would change this again:
Next recheck date:

This lets the stack age honestly.

⸻

Recheck And Cost Visibility

Cost may change faster than architecture.

Recheck cost when:

* token usage spikes
* fallback escalation appears
* subscription overlap grows
* API provider changes pricing
* local model becomes good enough
* cloud model becomes cheaper
* workflow volume increases

Stackfax rule:

Cost should leave a receipt.

⸻

Recheck And Context

Context should be rechecked when:

* memory grows
* old assumptions remain active
* stale docs are retrieved
* private data enters memory
* RAG results drift
* project/client boundaries blur
* full history keeps getting loaded

Stackfax rule:

Pass state, not history.

⸻

Recheck And Evaluation

A stack should be rechecked after real use.

Ask:

* Did the workflow repeat?
* Did quality hold?
* Did cost stay controlled?
* Did review burden drop?
* Did failures stay understandable?
* Did receipts help?
* Did the stack produce useful work?
* Did the user actually trust it?

Stackfax principle:

Benchmarks rank models. Workflow tests reveal fit.
