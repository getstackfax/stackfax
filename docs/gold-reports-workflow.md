# Stackfax Gold Reports Workflow

Gold Reports are Stackfax’s system for turning public Ai-stack chaos into durable product doctrine.

The goal is to capture the strongest public signals from Reddit, GitHub, YouTube, X, forums, comments, product launches, and builder discussions without spamming, scraping private data, or confusing field research with marketing.

Gold Reports are not content farming.

Gold Reports are field research.

⸻

Core Idea

Stackfax learns from repeated public confusion.

A Gold Report captures a high-signal example of that confusion and turns it into:

* doctrine
* risk flags
* intake questions
* report sections
* verdict language
* product lessons
* future website copy
* Reddit-safe reply patterns
* private engine notes

The raw thread is not the product.

The distilled lesson is the product.

⸻

Core Question

What does this public example reveal about how people actually struggle with Ai stacks?

A Gold Report should answer:

* What is the visible question?
* What is the hidden stack issue?
* What does Stackfax know now that it can reuse?
* Should this become doctrine, reply fuel, intake logic, or just field texture?

⸻

What Counts As A Gold Report

A public example may be Gold if it reveals a durable Stackfax pattern.

Gold examples often show:

* hardware confusion
* token burn confusion
* model subscription overlap
* OpenClaw setup confusion
* local vs cloud uncertainty
* business automation risk
* approval gates missing
* run receipts missing
* fragile UI automation
* credential isolation problems
* communication channel risk
* memory/context governance issues
* agent ROI unclear
* evaluation layer missing
* vendor stack overkill
* migration pressure after model/tool updates

Gold does not mean the post is perfect.

Gold means the pattern is useful.

⸻

Scouting Tiers

Gold

Highest-signal example.

Use when the public example can become doctrine, report logic, intake logic, or product positioning.

Gold becomes build material.

Silver

Useful pattern worth watching.

Use when the example is interesting but not strong enough to become doctrine yet.

Silver becomes watchlist.

Bronze

Low-confidence, repetitive, vague, promotional, or weak signal.

Use when the example shows market texture but should not drive product decisions.

Bronze becomes market texture.

Skip

Not worth preserving.

Use when the thread is too vague, too promotional, too hostile, too risky, or already fully answered.

Skip keeps the system clean.

Hidden Stack Issue

The hidden stack issue is the real Stackfax signal.

Examples:

Visible question:

Do I need a Mac mini?

Hidden stack issue:

Workflow is not defined yet. Hardware decision is premature.

Visible question:

Which model should I pay for?

Hidden stack issue:

Subscription overlap risk. Each paid tool needs a job.

Visible question:

My agent burned too many tokens.

Hidden stack issue:

Misrouted intelligence, context bloat, or missing budget caps.

Visible question:

Can I automate customer replies?

Hidden stack issue:

Business automation safety, approval gates, and customer data risk.

Visible question:

Should I copy this stack?

Hidden stack issue:

Vendor stack fit, overbuild risk, and workflow mismatch.

Keeper Lines

A keeper line is a short durable phrase that compresses the lesson.

Good keeper lines are:

* clear
* reusable
* specific
* memorable
* useful in reports or docs
* not dependent on the original thread

Examples:

* The model is not the stack.
* Hardware should match the job, not the hype.
* Agent output is a claim. A run receipt is evidence.
* Benchmarks rank models. Workflow tests reveal fit.
* Pass state, not history.
* Every paid tool should have a job.
* The channel is part of the stack.
* Process first. Automation second.
* Migration is not progress unless the workflow gets better.
* The stack is not the win. The useful output is the win.

Keeper lines can become doctrine.

* user preferences
* build-mode rules
* recurring strategic posture
* formatting preferences

Skip

Useful in the moment, but not worth preserving.

Core rule:

Project source the durable decision, not the temporary instruction.

Reply Decision Rule

Gold Reports do not require public replies.

Most scouting should not become a comment.

Before replying, ask:

* Is the thread actually asking for help?
* Can the reply be useful without a link?
* Can the reply stand on its own?
* Does the reply add a diagnostic frame?
* Would this look helpful if Stackfax were never mentioned?
* Is the community tolerant of project mentions?
* Is the account in cooldown?
* Is saving the thread better than replying?

Default:

Gold Reports first. Replies only when useful.

⸻

Reddit-Safe Reply Pattern

When replying, use a short helpful frame.

Pattern:

1. Acknowledge the useful part.
2. Name the hidden stack issue.
3. Give a practical decision frame.
4. Avoid pitching.
5. Exit cleanly.

Example:

The useful split here is workflow first, tool second.

I would map what the agent needs to do, what it can touch, what should require approval, and what would prove the workflow worked before adding more models or hardware.

Otherwise the stack can get bigger without getting more useful.

Do not link Stackfax unless directly invited and appropriate.

⸻

What Not To Do

Gold Reports should not become spam.

Do not:

* drop links randomly
* copy-paste promo replies
* DM people cold
* pretend to be neutral while promoting Stackfax
* scrape private information
* store customer/private data
* turn every thread into a pitch
* argue with hostile users
* chase every keyword
* overclaim results
* use fake testimonials
* imply Stackfax is finished when it is not

Field research should increase trust.

It should not create noise.

⸻

Public / Private Boundary

Gold Reports should use public information only unless a human explicitly provides approved material.

Allowed by default:

* public Reddit threads
* public GitHub repos
* public product pages
* public docs
* public YouTube videos or comments
* public X posts
* public forums
* public blogs

Not allowed by default:

* private DMs
* private emails
* customer submissions
* private screenshots
* private files
* credentials
* API keys
* payment data
* wallet information
* private business records
* personal account data

Core rule:

Public signal only unless explicitly approved.

30-Day Review

Gold Reports should be reviewed periodically.

Ask:

* Which patterns repeated?
* Which risk flags appeared most often?
* Which report types got stronger?
* Which keeper lines are worth promoting?
* Which old examples are stale?
* Which signals were noise?
* What should become a docs file?
* What should stay private engine logic?

Gold Reports should compound into doctrine.

They should not become a junk drawer.
