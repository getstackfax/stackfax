Stackfax Community Standards Plan

The Community Standards Plan defines which GitHub community files Stackfax may add later and when they should be added.

The goal is to make the repo more complete without accidentally giving away more rights, access, or expectations than intended.

⸻

Core Question

What community files should Stackfax add, and which ones should wait?

Community standards should support trust.

They should not accidentally redefine the business model.

⸻

Current Recommendation

Do not rush community standards before the repo’s role is clear.

Stackfax is not only code.

It includes:

* product positioning
* report logic
* scoring language
* badge language
* brand language
* templates
* marketing copy
* business structure
* future offer strategy
* public trust layer
* private business engine

Because of that, some community files are useful, but licensing should be handled carefully.

⸻

Files To Add Later

Code Of Conduct

Purpose

Sets basic expectations for respectful participation.

Recommended Timing

Add when people begin opening issues, commenting, contributing, or interacting with the repo.

Risk Level

Low.

Notes

The Code of Conduct should protect:

* beginners
* builders
* good-faith critics
* safety-focused discussion
* honest disagreement

It should reject:

* spam
* harassment
* fake testimonials
* unsafe automation advice
* credential sharing
* attempts to pressure the project into unsafe claims

⸻

Contributing Guide

Purpose

Explains how people can suggest changes, submit issues, or contribute improvements.

Recommended Timing

Add when outside feedback becomes useful.

Risk Level

Low to medium.

Notes

The contributing guide should make clear that not all ideas, wording, templates, report logic, badge ideas, or scoring suggestions will be accepted.

It should also make clear that contributions should not include private data, customer data, secrets, credentials, or fake testimonials.

Good early contributions:

* typo fixes
* clarity improvements
* report type suggestions
* badge ideas
* risk category ideas
* beginner feedback
* example scenario improvements
* documentation cleanup

Bad early contributions:

* private customer examples
* real customer reports
* API keys
* passwords
* payment details
* unsafe automation instructions
* broad rewrites of product doctrine without review

⸻

Security Policy

Purpose

Explains how to report security concerns.

Recommended Timing

Add before Stackfax handles user submissions, files, forms, customer data, customer reports, integrations, or automation.

Risk Level

Low.

Notes

A Security Policy is probably the safest first community file because Stackfax is a trust product.

It should say not to open public issues with sensitive details.

It should also state that the public repo should not contain secrets, private customer data, payment information, or credentials.

⸻

Issue Templates

Purpose

Helps people submit structured issues.

Possible Templates

* bug report
* documentation improvement
* report type suggestion
* badge suggestion
* risk category suggestion
* stack example suggestion
* safety concern
* unclear wording report

Recommended Timing

Add when public feedback starts.

Risk Level

Low.

Notes

Issue templates should warn users not to include:

* passwords
* API keys
* payment details
* wallet information
* customer data
* private business details
* sensitive personal information
* private screenshots

⸻

Pull Request Template

Purpose

Helps contributors explain what they changed and why.

Recommended Timing

Add when outside pull requests become likely.

Risk Level

Low.

Notes

The PR template should ask:

* What changed?
* Why does this help?
* Is this public-safe?
* Does this include any private data?
* Does this change report doctrine, scoring, badges, or product positioning?
* Should this be reviewed as public trust-layer content or private engine content?

⸻

File To Pause On

License

Purpose

Defines what others can legally do with the repo content.

Recommended Timing

Pause until Stackfax decides whether this repo is:

* open-source
* source-available
* public documentation only
* private-product foundation
* mixed public/private structure

Risk Level

High.

Why Pause

A normal open-source license may allow people to copy, reuse, modify, or commercialize parts of the Stackfax foundation.

That may be fine later, but it should be intentional.

Do not license away the engine before the business exists.

⸻

Community Standards Priority

Suggested order:

1. Security Policy
2. Code Of Conduct
3. Contributing Guide
4. Issue Templates
5. Pull Request Template
6. License decision

License should come after strategy, not before.

⸻

What The Repo Should Avoid For Now

Do not add community files that imply:

* Stackfax is fully open-source before that is decided
* anyone can freely reuse the full report system
* contributions are automatically accepted
* the repo is ready for production users
* the repo handles private user data
* Stackfax is already a finished product
* StackChecked is an official certification
* community examples are real customer results without permission
* public docs equal permission to clone the paid engine

The repo should stay honest about its phase.

⸻

Suggested Code Of Conduct Position

Keep it simple.

Stackfax should welcome:

* beginner questions
* builder feedback
* expert critique
* honest disagreement
* useful examples
* safety-focused suggestions
* public stack education
* workflow-first thinking

Stackfax should reject:

* spam
* harassment
* fake testimonials
* secret sharing
* credential sharing
* unsafe automation advice
* attempts to bypass approval gates
* misleading claims
* public sharing of sensitive private data
* pressure to overclaim results

⸻

Suggested Contributing Position

Contributions may be welcome for:

* typos
* clarity improvements
* report type suggestions
* badge ideas
* risk category ideas
* field guide improvements
* example scenario improvements
* beginner readability feedback
* public-safe documentation improvements

Contributions should not include:

* private user data
* real customer data without permission
* API keys
* passwords
* payment details
* personal identity documents
* unsafe automation instructions
* fake testimonials
* private screenshots
* customer reports
* proprietary third-party content

Contribution rule:

Public-safe suggestions only.

⸻

Suggested Security Position

If someone finds a security issue, they should not open a public issue with sensitive details.

They should contact the project through an approved private channel once one exists.

Until then, the repo should avoid storing anything sensitive.

Security policy should reinforce:

* no secrets in issues
* no credentials in examples
* no private customer data in public reports
* no private screenshots
* no payment details
* no API keys
* no wallet information
* no sensitive business records

⸻

GitHub Green Checklist

Community standards items GitHub may show include:

* description
* README
* Code of Conduct
* Contributing
* License
* Security Policy
* Issue templates
* Pull request template

The green checklist is useful, but it is not more important than product safety.

Do not chase green checkmarks blindly.

⸻

Recommended Near-Term Move

Add later, when ready:

* Security Policy
* simple Code of Conduct
* lightweight Contributing Guide

Pause:

* License decision

Do not add a broad open-source license just because GitHub suggests one.

⸻

Public / Private Boundary

Community files belong in the public trust layer.

They can explain:

* how to contribute safely
* how to report problems
* what not to share
* what kind of feedback helps
* what public docs are for

They should not expose:

* private report generation logic
* scoring engine details
* customer workflows
* private report prompts
* payment operations
* intake interpretation rules
* internal automation systems

Public trust layer.

Private business engine.

⸻

Stackfax Principle

Do not chase green checkmarks blindly.

Add community files when they help the project.

Pause on license until the product strategy is clearer.

⸻

Final Rule

Community standards should increase trust without giving away control.

Add the safety files first.

Decide the license only when the public/private boundary is firm.
