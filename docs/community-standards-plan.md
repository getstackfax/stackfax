# Stackfax Community Standards Plan

The Community Standards Plan defines which GitHub community files Stackfax may add later and when they should be added.

The goal is to make the repo more complete without accidentally giving away more rights, access, or expectations than intended.

---

## Core Question

What community files should Stackfax add, and which ones should wait?

---

## Current Recommendation

Do not rush community standards before the repo’s role is clear.

Stackfax is not only code.

It includes:

- product positioning
- report logic
- scoring language
- brand language
- templates
- marketing copy
- business structure
- future offer strategy

Because of that, some community files are useful, but licensing should be handled carefully.

---

## Files To Add Later

### Code Of Conduct

Purpose:

Sets basic expectations for respectful participation.

Recommended timing:

Add when people begin opening issues, commenting, contributing, or interacting with the repo.

Risk level:

Low.

---

### Contributing Guide

Purpose:

Explains how people can suggest changes, submit issues, or contribute improvements.

Recommended timing:

Add when outside feedback becomes useful.

Risk level:

Low to medium.

Notes:

The contributing guide should make clear that not all ideas, wording, templates, or report logic will be accepted.

---

### Security Policy

Purpose:

Explains how to report security concerns.

Recommended timing:

Add before Stackfax handles user submissions, files, forms, customer data, or integrations.

Risk level:

Low.

---

### Issue Templates

Purpose:

Helps people submit structured issues.

Possible templates:

- bug report
- report type suggestion
- badge suggestion
- risk category suggestion
- stack example suggestion
- documentation improvement

Recommended timing:

Add when public feedback starts.

Risk level:

Low.

---

### Pull Request Template

Purpose:

Helps contributors explain what they changed and why.

Recommended timing:

Add when outside pull requests become likely.

Risk level:

Low.

---

## File To Pause On

### License

Purpose:

Defines what others can legally do with the repo content.

Recommended timing:

Pause until Stackfax decides whether this repo is:

- open-source
- source-available
- public documentation only
- private-product foundation
- mixed public/private structure

Risk level:

High.

Why pause:

A normal open-source license may allow people to copy, reuse, modify, or commercialize parts of the Stackfax foundation.

That may be fine later, but it should be intentional.

---

## Community Standards Priority

Suggested order:

1. Security Policy
2. Code Of Conduct
3. Contributing Guide
4. Issue Templates
5. Pull Request Template
6. License decision

License should come after strategy, not before.

---

## What The Repo Should Avoid For Now

Do not add community files that imply:

- Stackfax is fully open-source before that is decided
- anyone can freely reuse the full report system
- contributions are automatically accepted
- the repo is ready for production users
- the repo handles private user data
- Stackfax is already a finished product

---

## Suggested Code Of Conduct Position

Keep it simple.

Stackfax should welcome:

- beginner questions
- builder feedback
- expert critique
- honest disagreement
- useful examples
- safety-focused suggestions

Stackfax should reject:

- spam
- harassment
- fake testimonials
- secret sharing
- credential sharing
- unsafe automation advice
- attempts to bypass approval gates
- misleading claims

---

## Suggested Contributing Position

Contributions may be welcome for:

- typos
- clarity improvements
- report type suggestions
- badge ideas
- risk category ideas
- field guide improvements
- example scenario improvements

Contributions should not include:

- private user data
- real customer data without permission
- API keys
- passwords
- payment details
- personal identity documents
- unsafe automation instructions
- fake testimonials

---

## Suggested Security Position

If someone finds a security issue, they should not open a public issue with sensitive details.

They should contact the project through an approved private channel once one exists.

Until then, the repo should avoid storing anything sensitive.

---

## GitHub Green Checklist

Community standards items GitHub may show include:

- Description
- README
- Code of Conduct
- Contributing
- License
- Security Policy
- Issue templates
- Pull request template

The green checklist is useful, but it is not more important than product safety.

---

## Stackfax Principle

Do not chase green checkmarks blindly.

Add community files when they help the project.

Pause on license until the product strategy is clearer.
